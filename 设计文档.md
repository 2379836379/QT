# 设计文档

## 1. 项目概述
项目想法来源：https://github.com/LeoDreamer2004/Never-Judge

NeverJudge 是一个面向 OJ 做题场景的桌面工具，提供基础编辑和提交能力，但其 IDE 功能较轻，且更偏向“写代码并提交”的流程。本项目希望换一个方向：不再把重点放在代码编辑器本身，而是使用 Qt 实现一个面向 OpenJudge 的任务管理应用。

项目目标站点：<http://openjudge.cn/>

本项目聚焦以下问题：

1. 题目分散在网页中，查找、筛选和跟踪完成情况不够方便。
2. 用户做题时通常需要自己维护“做了哪些题、哪些题值得重刷、哪些题还没看懂”等额外信息。
3. 网页端更适合提交，不适合长期管理刷题计划、笔记、标签和本地代码文件。

因此，本项目的定位是：

- 一个基于 Qt 的 OpenJudge 任务管理客户端；
- 负责题目浏览、状态同步、提交记录查看、本地笔记与标签管理；
- 不实现完整 IDE，不负责本地编译、调试、复杂代码编辑。

## 2. 项目目标与边界

### 2.1 项目目标
- 支持用户登录 OpenJudge 并维持会话状态。
- 支持浏览题目列表、题目详情和提交记录。
- 支持按题目状态、分类、关键词进行筛选。
- 支持为题目添加本地标签、备注、难度评价和复习状态。
- 支持管理本地题解/代码文件路径，方便从题目跳转到本地工程。
- 提供清晰的任务面板，帮助用户安排待做、已做、重做中的题目。

### 2.2 非目标
- 不实现完整代码编辑器。
- 不实现编译器、调试器或运行环境管理。
- 不自行维护题库镜像，题面内容以 OpenJudge 网站为准。
- 不保证替代网页端全部功能，复杂交互仍以浏览器为后备方案。

### 2.3 使用人群
- 主要用户：需要长期刷 OpenJudge 的学生或算法练习者。
- 次要用户：希望对题目做本地归档、整理标签和复盘笔记的用户。

## 3. 需求分析

### 3.1 功能需求

#### 3.1.1 账户与登录
- 用户输入账号密码登录 OpenJudge。
- 客户端保存会话 Cookie，在有效期内免重复登录。
- 登录失败时提示错误原因，如密码错误、验证码变化、网络异常。

#### 3.1.2 题目浏览
- 展示题目列表，包括题号、标题、来源、通过状态等信息。
- 查看题目详情，包括题面链接、限制信息、样例和题目分类。
- 支持按关键词搜索题目。
- 支持按题目来源、章节、是否已通过、本地标签等维度筛选。

#### 3.1.3 做题任务管理
- 为题目标记任务状态，如“未开始”“进行中”“已完成”“待重做”。
- 支持用户添加个人优先级和截止时间。
- 支持查看“今日待做”“最近未完成”“最近提交失败”等任务视图。

#### 3.1.4 本地信息管理
- 为题目添加标签，如“动态规划”“二分”“错题”“经典题”。
- 为题目添加个人笔记。
- 记录本地代码文件或题解文件路径。
- 可记录个人难度评分和掌握程度。

#### 3.1.5 提交记录查看
- 拉取用户在 OpenJudge 上的提交记录。
- 展示提交时间、语言、结果、运行时间、内存等信息。
- 支持按题目和结果筛选提交记录。
- 对某题显示最近一次 AC 时间和最近一次失败结果。

#### 3.1.6 同步与刷新
- 手动刷新题目列表与提交状态。
- 启动时加载本地缓存，避免每次都完整抓取。
- 对频繁访问页面做增量更新，减少请求量和等待时间。

### 3.2 非功能需求
- 跨平台：优先支持 Windows 与 Linux。
- 响应性：界面操作不能被网络请求阻塞。
- 稳定性：网络异常、站点页面变化时应有降级策略。
- 可维护性：UI、网络、解析、数据存储分层清晰。
- 可扩展性：后续可扩展到更多 OJ 或更多任务视图。

## 4. 总体设计

### 4.1 技术选型
- 框架：Qt 6
- UI 方案：Qt Widgets
- 网络：`QNetworkAccessManager`
- 持久化：SQLite + Qt SQL
- 配置管理：`QSettings`
- HTML 处理：基于 `QRegularExpression` + 结构化解析封装

选择 Qt Widgets 的原因：

- 更适合桌面工具型应用，表格、树状列表、对话框支持成熟；
- 和模型/视图架构结合后，题目列表与提交记录表格更容易组织；
- 对于本项目这类“信息管理工具”，Widgets 的开发效率和稳定性较高。

### 4.2 架构分层
系统采用分层设计：

1. 表现层（UI）
   负责窗口、对话框、题目列表、详情页、任务面板和用户交互。
2. 业务层（Service）
   负责登录流程、题目同步、提交记录同步、任务状态处理。
3. 数据访问层（Repository）
   负责 SQLite 读写、缓存更新、查询条件封装。
4. 网络与解析层（Gateway / Parser）
   负责访问 OpenJudge 页面并解析为结构化数据。

### 4.3 模块划分

| 模块 | 职责 |
| --- | --- |
| `ui` | 主窗口、登录窗口、题目列表页、题目详情页、任务面板 |
| `network` | HTTP 请求、Cookie 管理、超时和重试策略 |
| `parser` | 登录结果解析、题目列表解析、提交记录解析 |
| `service` | 业务编排，同步数据并驱动 UI 更新 |
| `repository` | 读写 SQLite，维护题目、本地标签、笔记、任务状态 |
| `model` | 题目、提交记录、任务、用户信息等数据对象 |
| `config` | 本地设置、缓存路径、用户偏好 |

## 5. 核心业务流程

### 5.1 登录流程
1. 用户输入账号密码。
2. 客户端向 OpenJudge 登录接口发起请求。
3. 网络层接收响应，解析是否登录成功。
4. 若成功，则保存 Cookie 和用户基本信息。
5. UI 跳转到主界面，并触发初次数据同步。

### 5.2 题目同步流程
1. 用户选择题库或分类视图。
2. 客户端请求对应题目列表页面。
3. 解析器提取题号、标题、来源、状态等信息。
4. Repository 将远程题目信息写入本地缓存。
5. UI 基于本地数据库刷新展示结果。

### 5.3 提交记录同步流程
1. 用户打开提交记录页面或点击刷新。
2. 客户端请求用户提交记录页面。
3. 解析器提取每条提交的题号、结果、时间和语言。
4. 将记录更新到本地数据库，并按题目汇总最近状态。
5. UI 刷新统计信息，如 AC 数、最近失败题、待重做题。

### 5.4 本地任务管理流程
1. 用户在题目详情页修改任务状态、标签、笔记或本地文件路径。
2. 修改写入 SQLite。
3. 题目列表与任务面板即时响应更新。
4. 本地数据不回写 OpenJudge，仅作为个人管理信息。

## 6. 数据设计

### 6.1 主要实体

#### 6.1.1 Problem
- `problem_id`：题号
- `title`：题目标题
- `source`：题目来源或分类
- `url`：题目页面链接
- `remote_status`：远端状态，如未提交、已通过
- `last_sync_time`：最近同步时间

#### 6.1.2 Submission
- `submission_id`：提交编号
- `problem_id`：所属题号
- `submit_time`：提交时间
- `language`：提交语言
- `result`：评测结果
- `time_cost`：运行时间
- `memory_cost`：内存占用

#### 6.1.3 ProblemNote
- `problem_id`
- `note_content`
- `difficulty_score`
- `mastery_level`
- `local_code_path`
- `updated_at`

#### 6.1.4 ProblemTask
- `problem_id`
- `task_status`：未开始、进行中、已完成、待重做
- `priority`
- `deadline`
- `review_flag`

#### 6.1.5 Tag / ProblemTag
- `tag_id`
- `tag_name`
- `problem_id`

### 6.2 数据库表设计

| 表名 | 说明 |
| --- | --- |
| `problems` | 题目基础信息 |
| `submissions` | 提交记录 |
| `problem_notes` | 题目笔记和本地附加信息 |
| `problem_tasks` | 任务状态与计划信息 |
| `tags` | 标签定义 |
| `problem_tags` | 题目与标签的多对多关系 |
| `users` | 当前登录用户缓存信息 |

### 6.3 缓存策略
- 题目基础信息以本地数据库缓存为主。
- 启动后优先展示缓存，再异步刷新远程数据。
- 提交记录按最近 N 页或最近 N 天增量同步。
- 对解析失败页面保留原始响应摘要，便于调试和兼容修复。

## 7. UI 设计

### 7.1 主界面布局
建议采用三栏式或上下分区布局：

- 左侧：分类导航、标签、任务视图入口；
- 中间：题目列表；
- 右侧：题目详情、个人笔记和本地文件信息。

主界面顶部可提供：

- 搜索框
- 筛选条件
- 刷新按钮
- 登录状态

### 7.2 主要页面

#### 7.2.1 登录窗口
- 输入用户名和密码
- 记住登录状态
- 显示错误信息

#### 7.2.2 题目列表页
- 表格列：题号、标题、状态、标签、优先级、最近提交结果
- 支持排序、筛选和搜索
- 支持双击打开题目详情

#### 7.2.3 题目详情页
- 显示题目基础信息与网页入口
- 显示最近提交记录
- 编辑个人标签、备注、任务状态
- 打开本地代码目录或文件

#### 7.2.4 提交记录页
- 按时间倒序展示
- 可筛选特定题目、语言、结果
- 支持跳转回对应题目详情

### 7.3 交互原则
- 所有网络操作异步执行。
- 列表优先秒开，详细信息按需加载。
- 对失败请求提供重试按钮和错误提示。
- 对本地编辑操作自动保存，减少额外确认步骤。

## 8. 关键实现方案

### 8.1 OpenJudge 访问方式
由于 OpenJudge 主要提供网页而非公开结构化 API，客户端需要以网页请求为主，核心包括：

- 模拟登录并保存 Cookie；
- 请求题目列表、题目详情、提交记录等页面；
- 将 HTML 页面解析为结构化数据。

为降低站点结构变化带来的风险，解析层需要：

- 统一封装选择器和正则规则；
- 对关键字段缺失做容错；
- 在解析失败时返回可诊断的错误信息。

### 8.2 异步刷新机制
- 使用 `QNetworkAccessManager` 发起异步请求；
- Service 层根据请求完成信号更新 Repository；
- Repository 变更后通知 UI 刷新模型；
- 长时间同步任务用进度提示或状态栏展示。

### 8.3 列表模型设计
题目列表和提交记录建议基于 Qt Model/View 架构实现：

- `QAbstractTableModel` 负责数据展示；
- `QSortFilterProxyModel` 负责排序与筛选；
- Repository 负责提供统一查询结果；
- UI 不直接持有数据库逻辑，减少耦合。

## 9. 异常与风险

### 9.1 风险点
- OpenJudge 页面结构变化可能导致解析失效。
- 登录流程若出现验证码、额外校验，自动登录复杂度会上升。
- 网络不稳定时可能出现同步中断或状态过期。
- 本地缓存与远端状态可能存在短暂不一致。

### 9.2 应对策略
- 将页面解析规则集中管理，降低修复成本。
- 对核心流程增加日志输出与失败提示。
- 保留“在浏览器中打开网页”的后备入口。
- 缓存和远端状态分开展示，避免误导用户。

## 10. 开发计划

### 10.1 第一阶段：最小可用版本
- 完成登录窗口与主窗口框架。
- 完成题目列表抓取与展示。
- 完成本地 SQLite 缓存。
- 支持题目标签、备注和任务状态维护。

### 10.2 第二阶段：增强功能
- 增加提交记录同步与统计视图。
- 增加复杂筛选和任务面板。
- 增加本地代码路径管理。
- 优化错误提示和同步体验。

### 10.3 第三阶段：可维护性提升
- 重构解析器和 Repository 接口。
- 增加日志系统与配置导出。
- 补充单元测试和解析回归测试。

## 11. 测试方案

### 11.1 功能测试
- 登录成功/失败流程测试
- 题目列表加载测试
- 提交记录同步测试
- 标签、备注、任务状态保存测试

### 11.2 异常测试
- 网络断开
- 会话过期
- 页面字段缺失
- 数据库写入失败

### 11.3 回归测试
- 对典型题目列表页和提交记录页保存样本 HTML；
- 页面解析逻辑修改后，对样本运行回归测试；
- 确保核心字段提取结果不被破坏。

## 12. 后续可扩展方向
- 接入更多 OJ 平台，形成统一题目管理器。
- 增加复习提醒与错题本视图。
- 增加题目统计图表，如周刷题量、AC 趋势。
- 增加云同步或导出功能。

## 13. 总结
本项目是一个以“任务管理”为核心、以 OpenJudge 为主要数据来源的 Qt 桌面应用。它不追求替代完整 IDE，而是专注解决刷题过程中的整理、跟踪、复盘和本地管理问题。

在实现上，项目应坚持“界面、业务、网络、存储分层”的设计原则，以本地缓存和异步刷新保证使用体验，并通过集中式页面解析与容错机制应对 OpenJudge 网页结构变化带来的风险。
